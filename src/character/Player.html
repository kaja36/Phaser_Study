Player.ts の解説

このファイルの役割

プレイヤーキャラクターの「見た目」と「動き」をまとめて管理するクラスです。 ScrollScene はこのクラスの update() を呼ぶだけで、 キー入力の判定やアニメーション切り替えの詳細を意識しなくて済むようになっています。

前提となる用語

Sprite(スプライト)
画像を表示できるPhaserのオブジェクトです。位置(x, y)・拡大縮小・回転・アニメーションなど、 画面に映る「絵」を動かすための基本パーツだと考えてください。 this.sprite はプレイヤーの見た目そのものを表しています。
Arcade Physics(アーケード物理)
Phaserに組み込まれている、シンプルな物理エンジンです。 重力・速度・衝突判定など「動き」に関する計算を代わりに行ってくれます。 後述の physics.add.sprite で作ったSpriteだけがこの機能を使えます。
body(物理ボディ)
Spriteに付属する、当たり判定や速度などの物理情報を持つオブジェクトです。 this.sprite.body でアクセスし、setVelocityX や body.touching.down のような物理に関する操作・参照はすべてこのbody経由で行われます。

各メソッドの説明と処理の流れ

このクラスには4つのメソッドがあります。呼ばれるタイミングと役割は次のとおりです。

① constructor(scene, x, y) — 生成時に1回だけ
ScrollScene.create() から new Player(this, PLAYER.startX, PLAYER.startY) として呼ばれます。
  1. createAnimations() を呼び、左向き/正面/右向きのアニメーションをScene側に登録する
  2. scene.physics.add.sprite(...) でプレイヤーの見た目(Sprite)を物理ボディ付きで作成し、this.sprite に保持する
  3. setBounce と setCollideWorldBounds で、跳ね返り方と画面外に出ない設定を行う
この時点ではまだ「動き」は始まりません。見た目と設定の準備だけが行われます。
② update(cursorKeys) — 毎フレーム呼ばれる
ScrollScene.update()(Phaserのライフサイクルで毎フレーム自動実行)から、 this._player.update(this._cursorKeys) として毎フレーム呼ばれます。
  1. 左キーが押されていれば、左方向に速度を設定し、左向きアニメーションを再生
  2. 右キーが押されていれば、右方向に速度を設定し、右向きアニメーションを再生
  3. どちらも押されていなければ、横方向の速度を0にし、正面向きアニメーションを再生
  4. 上キーが押されており、かつ地面に接地している場合のみ、上方向に速度を設定してジャンプさせる
左右の判定とジャンプの判定は独立しているため、「歩きながらジャンプ」も発生します。
③ tintDamage() — 爆弾に当たったときに1回
ScrollScene の handleHitBomb(プレイヤーと爆弾の衝突コールバック)から呼ばれます。 プレイヤーの色を赤くし(setTint)、正面向きアニメーションに固定します。 ゲームオーバー時の見た目変更だけを担当し、ゲームオーバーの判定やテキスト表示自体は ScrollScene 側の責務です。
④ createAnimations() — constructorから1回だけ呼ばれるprivateメソッド
外部からは呼べない内部処理です。1枚のスプライトシート画像から、 「左向き歩行」「停止(正面)」「右向き歩行」の3種類のアニメーションを切り出して Sceneに登録します。update() や tintDamage() は、 ここで登録した ANIMATION_KEYS のキーを指定して再生するだけです。
全体の流れ: ScrollScene.create() で Player が1つ生成され(① constructor)、 ゲームループが始まると毎フレーム ScrollScene.update() → Player.update()(②)が呼ばれて プレイヤーが操作されます。爆弾に当たった瞬間だけ ③ tintDamage() が割り込み、見た目が変化します。 ④ createAnimations() はその裏側で、①の最初に1回だけ実行される準備処理です。

TypeScriptの言語仕様

readonly プロパティ
readonly sprite: ArcadeSprite;
一度代入したら書き換えられないことを示します。 sprite はコンストラクタで1回だけ作られ、後から別のオブジェクトに差し替えることはない、という意図を表しています。
コンストラクタ引数の private readonly
constructor(private readonly scene: Phaser.Scene, x: number, y: number) {
引数に private readonly を付けると、 「this.scene = scene という代入」と「private readonly scene プロパティの宣言」を同時に行ったことになります(パラメータプロパティ)。 クラス内のどこからでも this.scene として使えます。
import type
import type { ArcadeSprite } from "../types/arcade.js";
型情報だけをインポートする書き方です。 ビルド後のJavaScriptには影響しない(型チェックのためだけに存在する)ことが明示されます。
private メソッド
private createAnimations(): void { ... }
クラスの外からは呼べないメソッドです。 アニメーション登録は Player 内部だけで完結する処理なので private にしています。

Phaser 4の仕様

physics.add.sprite
scene.physics.add.sprite(x, y, ASSET_KEYS.dude);
通常の add.sprite と違い、Arcade Physics(物理エンジン)用のbodyが付いたSpriteを作ります。 これにより setVelocity や setBounce など、物理に関するメソッドが使えるようになります。
setBounce / setCollideWorldBounds
this.sprite.setBounce(PLAYER.bounce);
this.sprite.setCollideWorldBounds(true);
setBounce は地面に着地したときの跳ね返り具合、 setCollideWorldBounds(true) は画面の端で止まる(外に出ない)ようにする設定です。
cursorKeys と update
update(cursorKeys: Phaser.Types.Input.Keyboard.CursorKeys): void
update() はSceneから毎フレーム呼ばれることを想定したメソッドです。 cursorKeys はキーボードの上下左右キーの押下状態を持つオブジェクトで、 cursorKeys.left.isDown のように現在押されているかを確認できます。
body.touching.down
if (cursorKeys.up.isDown && this.sprite.body.touching.down) {
  this.sprite.setVelocityY(PLAYER.jumpVelocity);
}
body.touching.down は「Spriteの下側が何かに接触しているか」を表します。 これを確認せずにジャンプ処理を書くと、空中にいてもジャンプキーを押すたびに上昇してしまい、 無限に空を飛べてしまいます。
anims.create と generateFrameNumbers
this.scene.anims.create({
  key: ANIMATION_KEYS.left,
  frames: this.scene.anims.generateFrameNumbers(ASSET_KEYS.dude, { start: 0, end: 3 }),
  frameRate: 10,
  repeat: -1,
});
1枚のスプライトシート画像(dude)の中から、指定した範囲のフレーム(0〜3番)を切り出して、 1つのアニメーションとして登録します。repeat: -1 はループ再生を意味します。
anims.exists による二重登録防止
if (!this.scene.anims.exists(ANIMATION_KEYS.left)) {
  this.scene.anims.create({ ... });
}
アニメーションはScene全体で共有されるリソースです。 Sceneが再起動された場合などに同じキーで anims.create を呼ぶとエラーになるため、 既に登録済みかどうかを anims.exists で確認しています。

設計のポイント: 継承ではなく「委譲」

Player クラスは Phaser.Scene や Phaser.GameObjects.Sprite を 継承していません。ただのTypeScriptクラスです。

Phaserの Scene クラスには、もともと update() というメソッドがあり、 ScrollScene extends Phaser.Scene でこれをオーバーライド(上書き)しています。 これはPhaserのライフサイクルの一部で、エンジンが毎フレーム自動的に呼び出します。

一方、Player.update() はPhaserとは無関係な、普通のメソッドです。 たまたま同じ名前を付けているだけで、Phaserから直接呼ばれることはありません。 ScrollScene.update()(Phaserが呼ぶ側)の中で、 

this._player.update(this._cursorKeys);

のように、ScrollSceneが自分の意思で Player の処理を呼び出しています。 このように「自分で処理せず、別のオブジェクトに処理を任せる」やり方を委譲(delegation)と呼びます。

継承だと「Player is a Scene(PlayerはSceneの一種)」という関係になってしまいますが、 実際にはそうではありません。委譲を使うことで、 「ScrollSceneはPlayerを持っている(has-a)」という自然な関係のまま、 Sceneの責務を肥大化させずにキャラクターの処理を分離できます。
