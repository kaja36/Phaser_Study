ScrollScene.ts の解説

このファイルの役割

このゲームの「舞台」となるSceneクラスです。 素材の読み込み(preload)、画面の初期構築(create)、毎フレームの更新(update)というPhaserの ライフサイクルに沿って、Player / Platforms / Stars / Bombs / ScoreBoard を組み立て、それぞれの間の衝突・接触判定を設定します。

ScrollScene自身はキャラクターの動きやアニメーションの詳細を持たず、 「誰と誰がぶつかったら何が起きるか」という全体の調整役に専念しています。

前提となる用語

Scene(シーン)
Phaserにおける「1つの画面・1つのゲーム場面」を表す単位です。 タイトル画面・ステージ1・ゲームオーバー画面などを、それぞれ別のSceneとして作ることもできます。 このプロジェクトでは ScrollScene という1つのSceneだけでゲーム全体を構成しています。
collider と overlap
どちらも2つのオブジェクト(またはGroup)の間で「重なり」を検出する仕組みですが、役割が異なります。
  • collider: 物理的にぶつかり合い、めり込まないように押し返す(地面とプレイヤーなど)
  • overlap: 押し返しは行わず、「重なったこと」だけを検出してコールバックを呼ぶ(プレイヤーとスターなど)
コールバック関数
handleCollectStar や handleHitBomb のように、 「衝突・接触が起きたときに呼ばれる関数」のことです。 arcade.html で解説した ArcadeCallback 型が使われています。

各メソッドの説明と処理の流れ

ScrollSceneは Phaser.Scene を継承しており、 Phaserのエンジンが決まったタイミングで preload → create → (update を毎フレーム繰り返す)の順にメソッドを呼び出します。 (詳しくは最後の「設計のポイント」で説明します。)

① constructor — Scene登録時に1回だけ
constructor() {
  super({ key: 'scrollScene' });
}
super(...) で親クラス(Phaser.Scene)の初期化を行い、 このSceneに 'scrollScene' という名前(key)を付けています。 フィールドの実際の値の代入はまだ行われません。
② preload — ゲーム開始前に1回だけ
画像・スプライトシートをすべて読み込みます。
  1. load.setBaseURL(ASSET_BASE_URL) で、以降の読み込み先のベースURLを指定する
  2. load.image(キー, パス) を4回呼び、sky/ground/star/bombの各画像を読み込む
  3. load.spritesheet(...) で、プレイヤー用のスプライトシート(複数フレームを含む画像)を読み込む
ここで読み込みが完了するまで、create() は呼ばれません。
③ create — ゲーム画面の初期構築(1回だけ)
  1. add.image(400, 300, ASSET_KEYS.sky) で背景画像を最初に追加する (Phaserは追加した順に後ろ→前へ重ねて描画するため、背景は最初に追加する)
  2. Platforms → Player → Stars → Bombs → ScoreBoard の順にインスタンスを生成し、各フィールドに保持する
  3. キーボード入力(cursorKeys)を初期化する
  4. physics.add.collider / physics.add.overlap を5回呼び、 オブジェクト同士の衝突・接触ルールを設定する
この時点で画面に必要なものはすべて配置され、衝突のルールも決まった状態になります。
④ update — 毎フレーム繰り返される
update(): void {
  if (!this._player || !this._cursorKeys) {
    return;
  }

  this._player.update(this._cursorKeys);
}
_player や _cursorKeys が未初期化の場合(create実行前など)は 何もせずに終了します。準備ができていれば、Player.update() を呼び出すだけです。 プレイヤーの移動・ジャンプ・アニメーションの判断は、すべて Player.html で解説している Player クラス側に委ねられています。
⑤ handleCollectStar — プレイヤーとスターが重なったとき
physics.add.overlap(player.sprite, stars.group, this.handleCollectStar, undefined, this) で登録されたコールバックです。
  1. 引数で渡された player / star を ArcadeSprite 型に変換する
  2. Stars.collect(starSprite) でそのスターを非表示にする
  3. ScoreBoard.add(SCORE.star) でスコアを加算する
  4. Stars.hasActiveStars() が false(全部回収済み)なら、 Stars.reset() でスターを再配置し、Bombs.spawnFromPlayer(playerSprite) で 新しいボムを出現させる
⑥ handleHitBomb — プレイヤーとボムが衝突したとき
physics.add.collider(player.sprite, bombs.group, this.handleHitBomb, undefined, this) で登録されたコールバックです。 physics.pause() で物理シミュレーション全体を停止し、 Player.tintDamage() でプレイヤーの見た目を変更し、 画面に「GAME OVER」のテキストを表示します。 引数の _player / _bomb はこの処理内では使われていません (先頭の _ は「あえて使っていない引数」であることを示す習慣的な命名です)。
全体の流れ: ①constructorでSceneが登録され、ゲーム開始時に②preloadで素材を読み込み、 ③createで画面とルールを構築します。その後はゲームが終わるまで④updateが毎フレーム繰り返され、 途中で衝突・接触が起きると⑤⑥のコールバックがその都度割り込んで実行されます。

TypeScriptの言語仕様

クラスの継承(extends)と super
class ScrollScene extends Phaser.Scene {
  constructor() {
    super({ key: 'scrollScene' });
  }
}
extends は「あるクラスの機能を引き継いで、新しいクラスを作る」ためのキーワードです。 super(...) は、引き継ぎ元のクラス(Phaser.Scene)のコンストラクタを呼び出します。 継承についての詳しい解説は、このページ最後の「設計のポイント」も参照してください。
! (非nullアサーション/確定代入アサーション)
private _player!: Player;
通常、クラスのプロパティはコンストラクタの中で値を代入しないとエラーになります。 しかし _player はコンストラクタではなく create() の中で代入される設計です。 型の末尾に ! を付けることで、「このプロパティは(コンストラクタ以外で)必ず代入されるので、 未代入チェックを省略してよい」とTypeScriptに伝えています。
アロー関数のクラスプロパティ
private handleCollectStar: ArcadeCallback = (player, star) => {
  ...
  this._stars.collect(starSprite);
  ...
};
handleCollectStar は「メソッド」ではなく、アロー関数を値として持つ プロパティとして定義されています。 通常のメソッド(handleCollectStar(player, star) { ... })を コールバックとして渡すと、呼ばれたときに this が ScrollScene を指さなくなることがあります。 アロー関数は定義された時点の this(=このScrollSceneインスタンス)を保持するため、 コールバックとして渡しても this._stars などが安全に使えます。
複数行のimport
import {
  ASSET_BASE_URL,
  ASSET_KEYS,
  ASSET_PATHS,
  PLAYER,
  SCORE,
  TEXT,
} from "../constants/game.js";
importする名前が多い場合、1行にまとめず複数行に分けて書くことができます。 動作は1行で書いた場合と同じですが、差分(diff)が見やすくなるなどのメリットがあります。

Phaser 4の仕様

preload / create / update というライフサイクル
Phaser.Scene を継承したクラスでは、preload / create / update という名前のメソッドを定義すると、Phaserのエンジンが自動的に 決まった順序・タイミングで呼び出してくれます(詳細は「設計のポイント」で解説します)。
load.setBaseURL / load.image / load.spritesheet
this.load.setBaseURL(ASSET_BASE_URL);
this.load.image(ASSET_KEYS.sky, ASSET_PATHS.sky);
this.load.spritesheet(ASSET_KEYS.dude, ASSET_PATHS.dude, { frameWidth: 32, frameHeight: 48 });
setBaseURL で指定したURLが、以降の load.image などのパスの先頭に付加されます。 load.image は1枚絵、load.spritesheet は 「同じサイズのフレームが並んだ1枚の画像」を読み込みます。 第3引数の { frameWidth, frameHeight } で1フレームのサイズを指定します。
add.image と描画順序
this.add.image(400, 300, ASSET_KEYS.sky);
画面に画像を追加します。Phaserは add した順番に、奥から手前へ重ねて描画します。 そのため、背景にしたい画像(sky)を最初に追加しています。
physics.add.collider / physics.add.overlap の引数
this.physics.add.overlap(this._player.sprite, this._stars.group, this.handleCollectStar, undefined, this);
引数は順に「対象1」「対象2」「コールバック関数」「processCallback(追加の条件判定、未使用ならundefined)」 「コールバック内でのthisとして使うオブジェクト」です。 最後に this(ScrollSceneインスタンス)を渡すことで、コールバック内で this._stars のようにScrollSceneのプロパティへアクセスできるようになります。
physics.pause()
this.physics.pause();
Arcade Physicsのシミュレーション全体を停止します。 これにより、ゲームオーバー後はプレイヤーもボムも動かなくなります (Scene自体やTextの表示処理は停止しないため、その後の this.add.text(...) は問題なく実行されます)。

設計のポイント: 継承(Scene) と 委譲(各クラス)の組み合わせ

ScrollScene は class ScrollScene extends Phaser.Scene という形で、 Phaserの Scene クラスを継承しています。 これは「ScrollScene is a Scene(ScrollSceneはSceneの一種である)」という関係です。

継承しているからこそ、preload / create / update という 特別な名前のメソッドを定義するだけで、Phaserのエンジンが自動的にそれらを呼び出してくれます。 これは Phaser.Scene 側にあらかじめ用意されている仕組み(ライフサイクル)に、 自分のクラスを「乗せる」イメージです。

これに対して Player / Platforms / Stars / Bombs / ScoreBoard は、いずれも Phaser.Scene や Sprite を継承していない、 ただのTypeScriptクラスです。ScrollSceneはこれらのインスタンスをcreate()で生成して保持し、 update() から this._player.update(...) のように 自分の意思で呼び出すことで処理を任せています。 これは継承とは異なる「委譲(delegation)」というやり方で、 Player.html の「設計のポイント」でも解説しています。

まとめると、このプロジェクトの設計は次のようになっています。

  • 継承: ScrollSceneだけがPhaser.Sceneを継承し、Phaserのライフサイクルに乗る
  • 委譲: ScrollSceneは、キャラクターや背景・UIなどの個別の処理を、各クラスのインスタンスに委譲する

こうすることで、Phaserに関する手続き(継承・ライフサイクル)はScrollSceneに集約されつつ、 ゲームの各要素(プレイヤー、足場、スコアなど)は独立したクラスとして読み書きしやすい状態を保っています。 他のプロジェクトでも、新しい要素を追加するときはこの「委譲」の形を踏襲すると、 ScrollScene(やそれに相当するSceneクラス)が肥大化しにくくなります。
